# MPC
The MPC contract currently serves two purposes:

  - Chain signatures — users submit a message to the contract, and the MPC network collaboratively produces a threshold ECDSA or EdDSA signature without any single node holding the complete private key.
  - Confidential key derivation (CKD) — users submit an ephemeral public key and a derivation path, and the MPC network produces an application-specific private key encrypted so that only the requester can unwrap it. The derived key is a BLS signature over BLS12-381, with the property that anyone can verify it was honestly derived from the network's master key without the master key ever being revealed.

In the case of TEE, we only depend on CKD. The benefit of CKD is that we can have entites running in TEE that will have access to the same `app_private_key` and possibly derive the same key-pair from it.

## MPC - CKD API
The CKD API of the MPC contract exposes three methods:

  - public_key(domain_id?) — view method that returns the master public key pk_mpc for a given domain.
  - request_app_private_key(derivation_path, app_public_key, domain_id) — the main entry point. The caller provides a derivation path and an ephemeral public key, and the MPC network responds asynchronously with an encrypted
  app_private_key. Accepts 1 yoctoNEAR deposit. Two variants of app_public_key are supported:
    - AppPublicKey(pk1) — single G1 point, privately verifiable variant
    - AppPublicKeyPV(pk1, pk2) — G1 + G2 pair, publicly verifiable variant
**NOTE** In both cases the only thing that can be verified is that the `app_private_key` was correctly derived from the master private key for a particular derivation path based on `env::predecessor_account_id` and `derivation_path`. 

```
SHA3-256("near-mpc v0.1.0 app_id derivation:" || account_id || "," || derivation_path)
```

In our case we should be fine with privately verifiable variant (TEE worker will verify the `app_private_key` on its own inside the TEE)

  - `derived_public_key(path, predecessor?, domain_id?)` — view method. For the **BLS12-381/CKD domain it simply returns pk_mpc** [ref](https://github.com/near/mpc/blob/45424d4cf743f86a2c570eecc6698ac2e8e9558e/crates/contract/src/lib.rs#L393) (so the same output as `public_key` view only method), since path-based derivation for CKD happens implicitly through the app_id computation inside request_app_private_key.
  ```
  let derived_public_key: dtos::PublicKey = match public_key {
      PublicKeyExtended::Secp256k1 { near_public_key } => {
          // ... derives child key from path
      }
      PublicKeyExtended::Ed25519 { edwards_point, .. } => {
          // ... derives child key from path
      }
      PublicKeyExtended::Bls12381 { public_key } => public_key,  // path ignored, returns pk_mpc as-is
  };
```


## APP_PRIVATE_KEY

TEE worker issues `app_private_key` request providing ephemeral public key that `app_private_key` should be encrypted with. The encrypted information is actually a point on the G1 group of the BLS12-381 elliptic curve (48 bytes compressed). That can be used to verify the fact that it was properly generated using the **bilinear pairing** property of BLS signatures.

the [docs](https://github.com/near/mpc/blob/45424d4cf743f86a2c570eecc6698ac2e8e9558e/crates/threshold-signatures/docs/confidential_key_derivation/confidential-key-derivation.md#L195) states that this G1 point should be used to get to the actual `app_private_key` value 

```
compute the key s = HKDF(sig), using a https://en.wikipedia.org/wiki/HKDF function.
```


## Encryption
At this point TEE worker is aware of `app_private_key` that in the original Outlayer implementation was used to derive individual encryption keys for each project.

```
HMAC-SHA256(master_secret, seed)        ← seed = "github.com/{owner}/{repo}:{account_id}"
```

Now project owner needs to encrypt secrets and provide them confidentially to TEE worker. In origin outlayer implementation it was used symetric encryption. Basically Outalyer was enforcing particular encryption key for the secretes derived from `app_private_key` and `project_id`. That means that it wasn't possible to rotate the secrets, and that there had to be a way to confidentially provide a symmetric encryption key to the project owner.


That could be solved with a better approach namely [ECDHE](https://en.wikipedia.org/wiki/Elliptic-curve_Diffie%E2%80%93Hellman). Basically the TEE worker could generate X25519 public key from `HMAC-SHA256(master_secret)`. And anyone could then provide secrets encrypted using an ephemeral keypair to the contract. Then the project owner knowing that public key (probably persisted on chain) can simply provide new secrets and TEE can accept an encrypted request based on `predecessor_id`.


## Alternative approach

As discussed before, alternatively we could generate a project-specific key-pair and maybe even expose it inside the WASM. Ideally we would want use derivation that allows for:
* be able to generate per project public key offline(A)
* in case of child private key beeing exposed only that single project credentials should be exposed(B)

[BIP-32](https://github.com/bitcoin/bips/blob/master/bip-0032.mediawiki) defines 2 common derivation schemes for effective describes 2 possible derivation schemes:

* Non-hardened:
```
  priv_child = tweak(pub_parent, path) + priv_parent
  pub_child  = tweak(pub_parent, path) · G + pub_parent
  pub_child  = priv_child · G

```

 Allows for deriving public key without revealing the private key. BUT **by having access to child_private_key and pub_parent anyone can recover parent private key**. For this reason its mainly used in wallets. Where single entity owns all parend and child private keys.

* Hardened:
```
  priv_child = tweak(priv_parent, path) + priv_parent
  pub_child  = priv_child · G
```

  It's ok to leak the child private key, but requires access to the master private key to derive the child private key

tldr;

```
  ┌─────────────────────────────────────────────────┬──────────────┬──────────┐
  │                                                 │ Non-hardened │ Hardened │
  ├─────────────────────────────────────────────────┼──────────────┼──────────┤
  │ Derive child pubkey from master pubkey(offline) │ yes          │ no       │
  ├─────────────────────────────────────────────────┼──────────────┼──────────┤
  │ Child privkey leak safe                         │ no           │ yes      │
  └─────────────────────────────────────────────────┴──────────────┴──────────┘
```

That means that if we want to achieve `child privkey leak safe` we need to give up on `derive child pubkey offline` that could be addressed by introducing additional registration step where after project is deployed, the worker needs to derive and provide the derived public key to the contract.

## Identity-Based Encryption (IBE)

> **Note:** This property is not mentioned anywhere in the MPC documentation and was discovered during investigation. A working demo can be found in [`crates/ckd-ibe-demo/src/main.rs`](https://github.com/near/intents/blob/review/outlayer/crates/ckd-ibe-demo/src/main.rs). The underlying algorithm is [Boneh-Franklin IBE (2001)](https://crypto.stanford.edu/~dabo/papers/bfibe.pdf).

This encryption is compatible with BLS signatures because **a BLS signature and an IBE private key are the same mathematical object** — both are computed as `msk · H(identity)`. This is not a coincidence: Boneh-Franklin IBE and BLS signatures were designed around the same bilinear pairing primitive and published the same year (2001). Any system that produces BLS signatures therefore automatically has IBE built in, for free. The CKD network produces `app_private_key = msk · H(pk_mpc || app_id)`, which is structurally identical to an IBE private key for the identity `(pk_mpc || app_id)` — so the encryption capability comes with it whether or not it is documented or intended.

Anyone knowing only `pk_mpc` and `app_id` can encrypt a message that only the holder of `app_private_key` can decrypt.

**Encrypt** — inputs:
- `pk_mpc` — master public key (from `public_key()` contract view method)
- `app_id` — derived from `account_id` and `derivation_path`
- `r` — randomly generated scalar (ephemeral, discarded after encryption)
- `message` — plaintext to encrypt

Outputs `(U, ciphertext)` — both sent to the recipient.

**Decrypt** — inputs:
- `app_private_key` — received from CKD
- `U` — ephemeral value from the ciphertext
- `ciphertext` — encrypted message

Outputs `plaintext`.

**The key advantage** is that the sender does not need to know the recipient's public key and does not need to coordinate with the recipient beforehand. Any party can encrypt to an `app_id` at any time — even before the TEE worker has registered any public key or is running at all. Only the legitimate holder of the `app_private_key` for that `(account_id, derivation_path)` can ever decrypt it.
